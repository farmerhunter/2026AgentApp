## 1. 设计背景：问题和需求

### 1.1 现实学习场景中的痛点

重点写“为什么需要这个系统”，避免一开始堆技术。

可展开内容：

- 初中学生每天产生大量学习材料：课本、作业、试卷、错题、课堂提醒、临时问题。
- 传统错题本只记录“错了什么”，很难记录“为什么反复错”和“下一步怎么练”。
- 学生复习时容易只看题目，不知道应该优先解决哪个问题。
- 家长和老师看到零散材料时，难以快速理解本周学习重点和风险。
- 普通聊天机器人通常只回答单次问题，不能持续整理学习证据和形成阶段性总结。

### 1.2 用户角色和需求

按角色写清楚需求：

- 学生：上传学习材料、确认重点题、获得少量具体行动建议。
- 家长：查看跨学科周报，理解薄弱点和跟进方向。
- 老师：参考重点题和问题模式，而不是只看分数。
- 评委：看到一个有任务规划、数据流、工具调用和反馈闭环的 AI 智能体应用。

- 评价维度对应：

  - 技术：结构化 contract、skill/prompt/job、可迁移架构。

## 2. 设计理念与核心创新

### 2.1 产品定位

核心表达：

- 本项目的核心不是错题本。
- 本项目的核心不是单轮问答机器人。
- 本项目也不只是资料归档工具。
- 本项目要构建的是把日常学习证据中的表面信息分析成“可追溯发现、学习记忆和下一步行动”的学习智能体。
- 这个学习智能体的重要理念是持续维护学习过程中的短期上下文和长期学习记忆，逐步形成对学生的深层认知：不仅知道学生这一次错了什么，也能看见一段时间内反复出现的状态、习惯、薄弱点和需要帮助的地方。

### 2.2 学习证据到行动的分层模型

需要作为文档重点展开。

推荐流程：

```text
学习证据 evidence
  -> 问题分析 problem analysis
  -> 局部发现 finding
  -> 待确定记忆 memory candidate
  -> 短期记忆 short-term memory
  -> 学生状态认知 student state model
  -> 聚合见解 insight
  -> 长期学习记忆 long-term learning memory
  -> 周报和行动建议
```

要解释每层含义：

- evidence：上传材料、切题结果、人工确认、备注、教材摘要。
- problem analysis：从题目、得分、错因备注和教材上下文中推断背后的知识、方法、审题或表达问题。
- finding：一次材料或一组证据产生的局部判断。
- memory candidate：可能值得记住的问题模式，但需要确认。
- short-term memory：近期上下文中的学习状态，例如本周重点题、最近备注、刚确认的问题和还没稳定下来的判断。
- student state model：基于短期上下文和长期学习记忆形成的学生状态认知，用来回答“这个学生现在最需要帮助的地方是什么”。
- long-term learning memory：经过确认或多次证据支持后保留下来的学习问题、习惯、偏好和有效帮助方式。
- insight：经过 consolidation 的阶段性见解。
- action：具体、少量、适合中学生执行的行动。

### 2.3 为什么强调“可解释的问题模式”

例子：

- 普通错题记录：反比例函数薄弱。
- 学习智能体记忆候选：遇到反比例函数图像题时，对 k 的正负和象限关系迁移慢。

要说明优势：

- 它把“错在哪里”的表层记录推进到“为什么会错”的深层分析。
- 更容易跨题目复用。
- 更适合形成行动建议。
- 对家长和老师更容易解释。
- 避免把单次错误永久贴标签。

### 2.4 人机协作原则

核心观点：

- OCR 和模型可以辅助处理材料。
- 学生/家长必须确认重点题和关键语义。
- 模型不应在证据不足时编造错因或趋势。

适合突出“中学生作者的工程判断”：知道 AI 的边界，并设计人工确认机制提高可信度。

### 2.5 拥抱 Vibe Coding 与全链路 AI 协作

本项目的另一个创新亮点，是主动拥抱新一代 AI 辅助开发方式。这里的 vibe coding 不是让 AI 代替作者完成作品，而是作者把 AI 和智能体作为协作工具，贯穿规划、设计、开发、部署验证和文档编写全过程。

可展开内容：

- 在规划阶段，用 AI 辅助拆解竞赛要求、用户需求、版本边界和发布路线。
- 在设计阶段，用 AI 辅助整理产品 brief、技术决策、API 设计、存储设计和智能体工作流。
- 在开发阶段，用 Codex、Claude Code 等编程智能体协助生成代码、检查实现、修复问题和维护 GitHub issue。
- 在部署验证阶段，用智能体协助执行 build、smoke test、VPS 部署检查和版本同步。
- 在文档阶段，用 AI 辅助把工程过程总结成技术文档、演示脚本和评审材料。

需要强调的判断：

- 作者负责提出目标、判断需求、确认范围和审查结果，AI 负责提高效率和补充实现能力。
- 这种开发方式本身也是 AI 应用实践的一部分，体现了中学生学习并使用新工具完成复杂工程的能力。
- 全链路 AI 协作显著提升了项目完整度：不仅做出界面，还形成了需求文档、架构设计、数据契约、版本管理、部署验证和竞赛文档。

## 3. 技术选型与智能体技术简介

本章用于向评委解释：本项目不是单纯做一个网页，也不是只调用一次大模型，而是把现代 Web 工程、结构化数据、智能体工作流、记忆机制和可部署架构组合起来，形成一个可以持续演进的学习智能体应用。

### 3.1 技术选型

建议从以下几个层面写清楚“为什么选这些技术”：

| 层面 | 选型 | 在项目中的作用 | 选择原因 |
| --- | --- | --- | --- |
| 前端框架 | Vite + React | 构建学习助手工作台、上传流程、切题确认、周报查看和执行模式切换 | React 适合组件化交互界面；Vite 启动和构建快，构建产物仍可作为静态文件部署 |
| 前端样式 | Tailwind CSS | 快速搭建一致的工作台式界面 | 适合中小型项目快速控制间距、颜色和响应式布局 |
| 数据格式 | JSON contract | 约束教材摘要、切题结果、人工确认、学习发现、周报等结构 | 前后端、sample data、job runner 和未来 API 可以共享同一数据契约 |
| 智能体能力定义 | skills + prompts | 把教材摘要、学习洞察更新、周报生成拆成任务型能力 | 让智能体行为可审查、可版本管理、可测试，避免只依赖临时聊天提示词 |
| 任务执行 | 智能体 job runner / job adapter | 触发学习洞察更新、周报生成，并返回任务状态和结果 | 适合表达“提交任务 -> 等待执行 -> 获取结果”的真实智能体工作流 |
| 执行模式 | static / api / real 可切换 | 演示阶段使用静态样例，真实能力阶段接 REST API 和 LLM | 降低发布风险，避免半成品 API 影响稳定演示，同时保留升级路径 |
| 后端 API | Node.js + Express REST API | 提供上传记录、人工确认、记忆决策、智能体 job 状态和结果查询 | 与前端同属 JavaScript 技术栈，开发和部署成本低 |
| 本地持久化 | SQLite | 保存结构化业务数据、job 状态、学习发现和记忆决策 | 单文件、低资源、易备份，适合 VPS 和竞赛阶段 |
| 大文件存储 | VPS 本地文件，后续可迁移 COS | 保存图片、PDF、题目 crop、OCR 原始结果等文件 | 当前阶段简单可靠；未来多用户真实数据量可迁移到私有对象存储 |
| 真实 AI 能力 | LLM API，例如 DeepSeek/OpenAI 同类模型 | 生成 findings、memory candidates、weekly insights 和 action candidates | 适合完成自然语言理解、归因、总结和建议生成 |
| 部署方式 | Nginx + systemd + VPS | 静态前端、API 服务、job runner 在服务器上运行 | 结构清楚、成本可控、便于 smoke test 和版本发布 |
| 工程协作 | GitHub issue / label / milestone / release | 管理阶段任务、版本边界和发布检查 | 避免多电脑、多 agent 开发时范围失控，保证可以明确判断何时发布 |

写作重点：

- 选型应服务于产品目标，而不是堆技术名词。
- 前端选择强调“可交互工作台”，不是营销页面。
- JSON contract 和 skill/prompt 是本项目最重要的工程基础，因为它们把“智能体输出”变成可验证、可展示、可迭代的数据。
- static/api/real 模式切换是发布风险控制手段：演示可以稳定，真实 API 能逐步接入。
- SQLite/Express/VPS 是务实选型；未来 PostgreSQL、对象存储和多用户权限是演进方向。

### 3.2 智能体技术简介

通俗地说，AI 智能体不是“问一句、答一句”的聊天机器人，而是一个围绕目标工作的软件系统。用户给它一个任务目标，它会结合已有上下文进行分析，必要时调用工具或外部 API，保存重要过程和结果，再把输出组织成可以继续使用的行动或数据。

一个典型智能体通常包含以下能力：

```text
目标和规则
  -> 感知输入
  -> 分析和规划
  -> 调用工具
  -> 生成结构化结果
  -> 记忆和反馈
  -> 下一轮更准确的行动
```

各部分含义：

- 目标和规则：人类先规定智能体要解决什么问题、不能做什么、输出格式是什么。本项目智能体的目标是帮助中学生从学习证据中发现问题模式和行动建议，不替代老师批改，也不凭空诊断学生。
- 感知输入：智能体读取用户上传的材料、OCR 结果、人工确认、文字备注、教材摘要和历史周报。在学习场景中，这些输入就是“学习证据”。
- 分析和规划：智能体不是简单复制输入，而是把表面现象继续推理成更深层的问题。例如“反比例函数图像题做错”只是现象，智能体还要分析背后可能是“参数符号与图像象限关系迁移慢”。
- 调用工具：智能体需要通过工具连接外部世界，例如读取 JSON、调用 OCR、查询数据库、触发 job、调用 LLM API 或生成周报文件。工具让智能体不只会说话，还能执行具体步骤。
- 结构化输出：真实应用不能只得到一段自由文本。学习智能体要把结果写成 findings、memory candidates、actions、weekly insights 等结构化数据，前端才能展示、确认、追踪和再次利用。
- 记忆和反馈：智能体需要记住近期上下文和长期模式。短期记忆帮助它理解本周刚发生的学习情况；长期学习记忆帮助它跨周识别反复出现的问题、偏好和有效帮助方式。
- 安全边界：智能体输出必须带证据来源和置信度，证据不足时应保留 unknown 或 uncertain，而不是把一次错误说成永久标签。

智能体和普通大模型问答的区别：

| 对比点 | 普通问答 | 智能体工作流 |
| --- | --- | --- |
| 目标 | 回答当前问题 | 围绕长期目标持续推进任务 |
| 输入 | 当前一轮对话 | 当前输入 + 历史上下文 + 工具结果 + 记忆 |
| 输出 | 通常是一段文字 | 可展示、可验证、可存储的结构化结果 |
| 行动能力 | 主要是生成文本 | 可以调用 OCR、数据库、API、job runner 等工具 |
| 记忆能力 | 多数只依赖当前上下文 | 可以维护短期上下文和长期学习记忆 |
| 适合场景 | 临时解释、答疑 | 多步骤任务、持续跟踪、个性化支持 |

放到本项目中，学习智能体的核心价值可以概括为三句话：

1. 它把学习材料从“零散文件”变成“可分析的学习证据”。
2. 它把表面错误从“这道题错了”推进到“背后是什么问题模式”。
3. 它通过短期上下文和长期学习记忆，在持续使用中越来越了解学生的状态和需要帮助的地方。

因此，本项目的设计重点不是让模型一次性说出漂亮总结，而是建立一条可持续工作的智能体链路：

```text
学习证据
  -> 智能体 skill 分析
  -> 工具/API/job runner 执行
  -> 结构化 findings 和 memory candidates
  -> 人工确认和记忆更新
  -> 学生状态认知
  -> 周报 insight 和下一步行动
```

这样的设计更适合中学生学习场景。因为学习问题通常不是一次材料就能判断清楚，而是需要不断积累证据、保留不确定性、让学生确认关键语义，并在多周使用中逐步形成稳定认知。

### 3.3 主流智能体方案和选择

当前主流智能体方案大致可以分成几类：

| 方案类型 | 代表方案 | 适合场景 | 对本项目的启发 |
| --- | --- | --- | --- |
| 通用智能体 SDK | OpenAI Agents SDK、Microsoft Agent Framework / AutoGen | 需要标准化 agent、tool、handoff、tracing 或多智能体协作的应用 | 说明智能体工程正在从“提示词”走向“可观测、可组合、可调用工具”的软件架构 |
| 图式工作流和状态机 | LangGraph / LangChain | 需要复杂分支、持久状态、human-in-the-loop 和多步骤推理的应用 | 学习智能体也需要把 evidence、finding、memory、insight 等状态显式建模 |
| 多角色任务编排 | CrewAI 等 | 需要多个角色分工完成研究、写作、执行和审核的任务 | 可借鉴角色拆分思想，但本项目核心是一个学习智能体围绕学生状态持续工作 |
| 低代码智能体平台 | Dify、Coze 等 | 快速搭建聊天机器人、知识库问答、工作流应用 | 适合快速验证想法，但竞赛技术文档需要展示自有数据契约、工作流和部署控制 |
| 自建轻量运行时 | 项目内 Hermes runtime | 需要完全贴合项目数据结构、演示方式、版本边界和部署条件 | 可以最清楚地展示智能体如何从学习证据生成结构化记忆和行动 |

本项目选择 Hermes 作为运行时智能体引擎，而不是把 Hermes 当成产品本身。产品面向用户时是“学途智伴学习智能体”；Hermes 是背后的 engine/runtime，负责承载 skill、prompt、job runner、执行模式切换和结果落盘。

选择 Hermes runtime 的原因：

- 数据契约可控：项目已有教材摘要、切题结果、人工确认、学习发现、周报等 JSON contract，自建 runtime 能严格围绕这些 contract 设计。
- 演示稳定：竞赛提交需要稳定展示，Hermes 可以支持 static fixture、API job 和 real LLM 多种模式切换。
- 工作流透明：评委可以看见智能体如何从输入到 finding、memory candidate、student state、weekly insight，而不是只看到一个黑盒平台回答。
- 成本和部署可控：VPS、Express、SQLite、job runner 的组合足够支撑竞赛和早期真实 API 阶段。
- 方便学习和解释：自建轻量运行时能展示作者对智能体架构、工具调用、记忆和人机协作的理解。

因此，后续章节在描述用户价值、功能、算法和工作流时应尽量使用“学习智能体”“本智能体”“智能体运行时”等表达；只有在说明具体工程实现、文件名、配置项和运行时选择时才使用 Hermes。

### 3.4 开发阶段 AI 和智能体

本项目不仅在产品中构建学习智能体，也在开发过程中全链路使用 AI 和智能体协作。这一点体现了对新工具和新开发方式的主动学习。

开发阶段的 AI/智能体协作可以分为几层：

- 规划智能体：辅助拆解竞赛要求、版本路线、阶段任务、GitHub issue 和 release checklist。
- 设计智能体：辅助生成和审查产品 brief、技术决策、存储设计、API 设计、智能体运行时设计和部署文档。
- 编程智能体：Codex、Claude Code 等负责在明确任务边界内阅读代码、修改实现、运行验证、总结风险。
- 应用智能体：Hermes runtime 负责项目内学习场景的教材摘要、学习洞察更新、周报生成等任务。
- 验证智能体：辅助执行 build、数据校验、smoke test、部署检查和本地/远端同步确认。
- 文档智能体：辅助把开发过程整理成技术文档、演示脚本、答辩材料和可引用设计依据。

这种协作方式的关键不是“把任务交给 AI”，而是建立清晰的人机分工：

- 作者决定方向、目标、用户价值和版本边界。
- AI 帮助快速生成候选方案、发现遗漏、补充实现和整理文档。
- 作者通过 GitHub issue、设计文档、数据契约、测试和部署验证来约束 AI 输出。
- 多 agent 并行开发时，通过 milestone、label、release checklist 和版本分支控制范围，避免功能越界。

可在正式文档中强调：本项目本身就是一次完整的 AI 应用实践。它不仅应用智能体服务中学生学习，也把 AI 工具用于工程创作全过程，从而提升效率、完整度和可验证性。

## 7. 核心算法与工作流

### 7.1 证据归一化算法

输入：

- 教材摘要。
- 上传元数据。
- 切题结果。
- 人工确认。
- 文字备注。
- 历史周报或记忆。

输出：

- 统一 evidence 结构。
- source refs。
- subject / subject_label。

### 7.2 表面信息到深层问题的分析链路

这是智能体在工作流中的核心作用，应作为本章重点说明。本智能体不是把材料简单分类，而是把可见信息逐步抽象成问题模式：

```text
题目文本 / 得分 / 人工备注 / 教材上下文
  -> 表面现象：做错、不会、漏写、表达不完整
  -> 中间判断：知识点、步骤、审题、表达、记忆或习惯问题
  -> 深层问题模式：可解释、可追溯、可被后续证据修正
  -> 学生状态认知：把近期证据和长期学习记忆结合，判断学生当前最需要支持的地方
  -> 行动建议：针对问题模式设计练习或复习任务
```

可写分析方法：

- 对题目文本和教材摘要做概念关联，判断问题属于哪个学习单元。
- 对得分、错因备注和标签做证据加权，区分“知识点不会”和“会但迁移慢”。
- 对同类 finding 做聚合，识别是否是重复出现的问题模式，并把稳定模式写入长期学习记忆。
- 把近期上下文和长期学习记忆一起用于判断学生状态，避免每次分析都像第一次见到这个学生。
- 对证据不足的情况保留不确定性，不强行给出深层结论。

举例：

- 表面信息：反比例函数图像题做错。
- 中间分析：题目涉及 k 的正负与象限关系，学生备注中提到“分不清”。
- 深层问题：对参数符号和图像象限之间的规则迁移慢。
- 行动建议：整理 k 正负与象限对应表，并完成 2 道同类变式题。

### 7.3 知识点关联算法

目标：

- 把重点题、备注和教材章节建立关联。

规则：

- 明确匹配则记录 concept links。
- 不确定则标注 uncertain。
- 不强行映射所有问题。

### 7.4 错因推理与问题分类

建议分类：

- concept_gap
- procedure_gap
- calculation_error
- reading_comprehension
- expression_issue
- memory_recall
- carelessness
- study_habit
- unknown

说明：

- 不同学科允许不同问题类型。
- 证据不足时输出 unknown。
- 分类结果不是最终结论，而是从表层证据走向深层问题模式的中间步骤。

### 7.5 待确定记忆生成规则

这里的“记忆”不是简单保存原始材料，而是智能体对学生学习状态的持续认知。短期记忆保留近期上下文，长期学习记忆保留经过确认或多次证据支持的问题模式、学习习惯和有效帮助方式。两者共同让学习智能体在持续使用中越来越了解学生。

进入 memory candidate 的条件：

- 同类问题重复出现。
- 学生明确标注不理解。
- 属于影响后续学习的基础知识点。
- 老师或家长强调。
- 本周多次出现。

不进入长期记忆的条件：

- 单次偶发错误。
- 证据不足。
- 无法确认学科或知识点。

### 7.6 行动候选生成规则

行动类型：

- 重做题目。
- 复习概念。
- 回看教材章节。
- 整理笔记。
- 做专项练习。
- 向老师/家长/智能体追问。
- 下次复查。

原则：

- 数量少。
- 具体可执行。
- 和 evidence/finding 有关联。
- 适合中学生时间和认知负担。
- 优先针对深层问题模式，而不是只针对某一道题。

### 7.7 周报 consolidation

输入：

- 本周 findings。
- 短期记忆。
- 长期学习记忆。
- 重点题。
- 备注。
- 教材摘要。

输出：

- overall summary。
- subject summaries。
- main risks。
- student state updates。
- suggestions。
- next actions。

强调：

- insight 必须经过 consolidation。
- 周报不重新确认题目是否重要。
- 不从证据不足材料中编造趋势。
- 周报的价值在于把多个局部发现进一步合并成阶段性问题主线，而不是罗列本周材料。
- 周报也是更新学生状态认知的节点，用本周证据修正或强化长期学习记忆。

## 12. 开发过程与中学生创新实践

### 12.1 从需求识别到系统设计

讲述作者如何从学习生活问题出发：

- 观察错题本和周报痛点：许多记录停留在“错了哪道题”，没有继续分析“为什么错”和“下次如何避免”。
- 发现真正需要解决的不是材料收集，而是从表面学习信息中挖掘深层问题。
- 进一步把智能体理念落到学习场景：通过短期上下文和长期学习记忆持续构建学生状态认知，让系统越来越了解学生，而不是每次都从零开始分析。
- 把“学习助手”拆成多个可演示工作流。
- 把智能体的核心任务定义为分析链路：证据归一化、知识点关联、错因推理、问题模式提炼、行动建议生成。
- 设计多学科字段和跨学科周报。
- 通过人工确认机制保证深层分析有可信证据来源，避免模型凭空下结论。

### 12.2 利用 AI 和智能体辅助开发的方式

需要体现“中学生利用新技术创新的能力”，但不要写成完全由 AI 代做。

可写：

- 使用 AI 辅助梳理需求、竞赛规则和设计文档。
- 使用编程智能体协助实现前端、后端、数据契约和部署脚本。
- 使用项目内学习智能体验证教材摘要、学习洞察更新、周报生成等核心工作流。
- 人工审查、调整范围、控制隐私和发布边界。
- 通过 GitHub issue、label、milestone 和 release checklist 管理多 agent 协作任务。
- 使用版本计划管理 v1/v2/v3，避免 AI 生成内容突破当前发布范围。

### 12.3 工程实践与关键取舍

本节简要说明项目不是一次性写页面，而是按工程流程推进。重点写三件事即可：

- 设计先行：用产品 brief、ADR、API/存储设计和 JSON contract 先把边界定清楚，再进入实现。
- 验证闭环：通过 demo data 校验、前端 build、job runner smoke test 和 VPS 部署检查保证结果可运行。
- 控制范围：先做稳定闭环，再接真实 API；保留人工确认，不盲目相信 OCR/LLM；用版本计划避免半成品功能影响正式发布。

## 13. 应用价值与创新亮点

### 13.1 对学生的价值

- 把材料整理成可复习行动。
- 减少不知道从哪里复习的问题。
- 帮助形成学习反思。
- 让学习助手在持续使用中逐步了解自己的薄弱点、学习习惯和需要帮助的方式。

### 13.2 对家长和老师的价值

- 快速了解本周学习情况。
- 看到证据来源。
- 关注具体问题模式和行动，而不只是分数。
- 看到学生状态如何随时间变化，哪些问题正在改善，哪些问题需要继续支持。

### 13.3 技术创新

- 分层语义模型：evidence -> finding -> memory -> student state -> insight -> action。
- task-specific skills。
- static demo job adapter。
- 数据契约驱动前后端。
- 可在静态演示和真实 API 之间平滑切换。
- 短期上下文与长期学习记忆结合，使智能体具备持续理解学生状态的能力。
- 全链路 AI 协作开发：规划、设计、编码、部署验证和文档编写都纳入可审查的工程流程。

### 13.4 体验创新

- 工作台式 UI。
- 上传结束页直接承接学习发现。
- 待确定记忆确认。
- 跨学科周报。

## 14. 不足与改进方向

## 16. 正式写作建议

### 16.1 叙事主线

建议正式技术文档围绕这一条线展开：

```text
真实学习问题
  -> 智能体设计理念
  -> 可解释学习记忆模型
  -> Web 工作台和数据契约实现
  -> 稳定全链路演示
  -> 真实 AI/API 和多用户演进
```

### 16.2 需要重点突出的亮点

- 利用AI开发AI能力
- 不做泛泛聊天机器人，不做简单错题本。目标是做“越来越懂你”的长期助手
- 保留人工确认，尊重 AI 边界。
- 用结构化 contract 约束模型输出。
- 用版本化 skill/prompt/job 体现智能体工程化。
- AI的泛化能力带来的未来强扩展能力

### 16.3 建议配图

- 系统总体架构图。
- 学习证据到行动建议的流程图。
- 上传材料到学习发现的工作流图。
- 周报 consolidation 流程图。
- v1/v2/v3 演进路线图。
- Web UI 主页面截图和上传流程截图。

### 16.4 正文篇幅建议

如果比赛没有严格页数限制，建议：

- 摘要：0.5 页。
- 背景和价值：1-1.5 页。
- 功能和工作流：2-3 页。
- 技术架构和算法：3-5 页。
- 工具调用和数据契约：2-3 页。
- 测试部署和隐私：1-2 页。
- 创新亮点与改进：1-2 页。

### 16.5 写作风格要求

正式成文时要尽量避免明显的“AI 味”。文字应简单、清楚、具体，像作者自己解释项目，而不是堆砌概念。

具体要求：

- 少用长而稀疏的列表。能用一段话讲清楚的，就不要拆成十几条 bullet。
- 每个列表只保留真正有信息量的点，避免“提高效率、提升体验、增强能力”这类空泛表达。
- 多写真实取舍和具体例子，例如为什么保留人工确认、为什么用 JSON contract、为什么先做 static/api 切换。
- 少用宏大口号，多写项目里真实发生的流程、文件、数据和验证方式。
- 句子尽量短，段落不要太长。评委应能快速看懂“问题是什么、智能体做了什么、为什么这样设计”。

总长度可控制在 12-18 页左右，配图后更易读。
