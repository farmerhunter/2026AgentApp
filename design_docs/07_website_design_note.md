# Hermes Web 网站草图设计笔记

本文档用于记录开发者对 Hermes Web 页面、功能和页面关系的初步设计。它不是最终 UI 规范，而是给 Codex 或前端开发者使用的文本草图。

## 1. 网站目标

Hermes Web 第一版不是营销首页，而是一个可操作的学习助手工作台。

核心任务：

1. 在“学习成果”中按学科管理作业、试卷、练习册页面和错题本照片。
2. 通过“上传材料”子流程上传材料、指定学科、查看腾讯云试卷切题结果、确认重点题。
3. 在“输入备注”中按学科补充课堂提醒、自己不理解的知识点和复习备注。
4. 在“历史周报”中查看 Hermes 生成的跨学科周报、各学科学习总结和建议。
5. 在“学习内容”中按学科查看课本、讲义或参考资料的理解结果。

## 2. 用例到页面和视图的映射

本章按照 `design_docs/03_design_brief.md` 第 3 章的第一版核心体验用例组织。Hermes Web 第一版建议采用单页应用，但页面设计应优先服务这些用例，而不是先从页面数量出发。

顶层导航统一为四项：

```text
Hermes Web 学习助手工作台
  -> 学习成果
  -> 输入备注
  -> 历史周报
  -> 学习内容
```

学科不作为顶层导航。第一版默认支持 `语文 / 数学 / 英语` 三个学科，并在相关视图中提供 `全部 / 语文 / 数学 / 英语` 的筛选器或内容分区。内部数据建议统一使用 subject code：

```text
chinese = 语文
math = 数学
english = 英语
```

用例与 Web 承载关系：

```text
用例 1：上传课本 PDF，建立学习内容理解
  -> 学习内容视图

用例 2：上传学习成果并标注重点题
  -> 学习成果视图
  -> 上传材料子流程
  -> 重点题确认步骤

用例 3：输入文字形式的知识点和备注
  -> 输入备注视图

用例 4：生成重点题记录并更新记忆
  -> 上传材料子流程的保存结果
  -> 学习成果视图中的重点题摘要
  -> 后续 Hermes job / memory update 状态

用例 5：生成阶段性学习分析和周报
  -> 历史周报视图中的生成状态、周报详情和行动建议

用例 6：Web 前端查询和展示周报
  -> 历史周报视图
```

### 2.1 用例 1：学习内容视图

援引 `design_docs/03_design_brief.md`：用例 1 是“上传课本 PDF，建立学习内容理解”。

Web 承载方式：

- 顶层导航项：`学习内容`
- 第一版应支持上传课本、讲义或教材 PDF，并触发 Hermes 分析。
- 上传或登记学习内容时必须指定学科。
- Hermes 分析完成后，将目录、章节、学习单元和知识点保存为教材摘要 JSON。
- 为控制范围，第一版可以只支持单个脱敏样例 PDF 或少量页码范围，不要求完整处理所有教材版式、扫描质量和复杂目录结构。
- 该视图为后续学习成果、重点题和周报提供先验学习上下文。
- 该视图需要提供学科筛选或学科分区：全部、语文、数学、英语。

功能：

- 上传课本、讲义或教材 PDF。
- 选择资料所属学科。
- 显示上传和 Hermes 分析状态。
- 保存并展示教材摘要。
- 展示课本、讲义或参考资料的理解结果。
- 展示章节、学习单元和知识点。
- 展示与本周学习成果的关联提示。

主要内容：

- 学科筛选器或学科分区。
- PDF 上传区域。
- 资料信息表单：学科、年级、资料标题、页码范围或章节范围。
- Hermes 分析状态：待上传、分析中、已完成、失败。
- 教材或资料摘要。
- 章节列表。
- 学习单元列表。
- 关键知识点。
- 与作业、试卷、备注和重点题的关联入口。

输出：

- 教材摘要保存为 `textbook_content_summary` contract 形状。
- 本地 demo 阶段可以写入或模拟写入 `/data/textbooks/{textbook_id}/textbook_content_summary.json`。

### 2.2 用例 2：学习成果视图、上传材料子流程和重点题确认步骤

援引 `design_docs/03_design_brief.md`：用例 2 是“上传学习成果并标注重点题”。

Web 承载方式：

- 顶层导航项：`学习成果`
- 子流程：`上传材料`
- 子步骤：`重点题确认`
- 学习成果列表和重点题摘要需要支持按学科筛选或分区。

“学习成果”视图是材料和结果的汇总入口。上传材料、试题分割和重点题确认不是独立顶层导航，而是从“学习成果”进入的子流程。

#### 2.2.1 学习成果视图

功能：

- 展示已添加的作业、试卷、练习册页面和错题本材料。
- 按学科过滤或分组展示学习成果材料。
- 展示上传状态、OCR / 切题任务状态和最近处理结果。
- 展示从材料中产生的重点题确认状态。
- 提供“上传材料”按钮，进入上传材料子流程。

主要内容：

- 学科筛选器：全部、语文、数学、英语。
- “添加材料”区域：说明当前可以添加哪些学习成果材料。
- “上传材料”按钮：启动具体上传和处理流程。
- 最近材料列表。
- 处理状态：待上传、识别中、待确认、已确认、已进入周报。
- 重点题摘要：已选重点题数量、待复习数量、关联周报。

命名约定：

- 区域标题使用“添加材料”，表示这是学习成果页中的材料入口区域。
- 主要按钮使用“上传材料”，表示点击后会进入真正的上传和处理流程。
- 避免同时把区域标题和按钮都写成“添加材料”，以免用户无法区分信息区和动作。

#### 2.2.2 上传材料子流程

“上传材料”不是顶层主视图，而是从“学习成果”视图进入的子流程。

推荐第一版采用工作台内的全屏子页面或 route-like 子页面，而不是把全部复杂交互内嵌在学习成果列表中。原因是上传、试题分割、原图预览、题目框叠加和重点题确认需要较大的操作空间。

如果后续只上传一张图片并填写少量字段，可以考虑 modal dialog；但第一版更适合全屏子页面。

流程步骤：

```text
学习成果
  -> 点击“上传材料”
  -> 上传材料子流程
     1. 选择文件和填写材料信息
     2. 提交识别或读取 demo 切题结果
     3. 查看原图和题目框
     4. 选择需要记录的重点题
     5. 补充得分、知识点、错因、备注和复习优先级
     6. 保存确认结果
  -> 返回学习成果视图
```

主要内容：

- 文件选择区域。
- 学生、学科、来源类型、日期等基础信息。
- 上传和识别进度。
- 原图预览。
- 题目框叠加。
- 切题列表。
- 重点题确认表单。
- 操作按钮：取消、保存草稿、保存确认结果、返回学习成果。

#### 2.2.3 重点题确认步骤

重点题确认属于“上传材料子流程”中的一个步骤，不作为顶层独立导航。

功能：

- 展示原始试卷图片。
- 在原图上叠加题目框。
- 展示切题列表。
- 选择需要记录的题。
- 为选中题填写得分、满分、知识点、错因、备注和复习优先级。
- 保存确认结果。
- 触发或等待 Hermes 总结。

主要内容：

- 左侧或主区域：原图预览和题目框。
- 右侧或下方：题目列表。
- 单题卡片：题目图片、识别文字、选择状态、确认表单。
- 操作按钮：保存确认结果、生成总结、返回学习成果、进入周报。

### 2.3 用例 3：输入备注视图

援引 `design_docs/03_design_brief.md`：用例 3 是“输入文字形式的知识点和备注”。

Web 承载方式：

- 顶层导航项：`输入备注`
- 第一版可以只做本地 demo 保存状态，不要求真实后端写入。
- 备注保存时必须指定学科；默认值可以沿用最近一次选择的学科。
- 保存后的备注后续应进入 Hermes 分析和周报上下文。

功能：

- 输入课堂重点、老师提醒、学习问题和复习备注。
- 选择学科、主题或关联章节。
- 保存备注，并加入本周学习上下文。
- 查看最近备注。

主要内容：

- 文字输入框。
- 学科、主题、日期等基础字段。
- “保存备注”按钮。
- 可选的“加入本周上下文”按钮或开关。
- 最近备注列表。

设计要求：

- 输入框不能单独出现，必须有明确操作按钮。
- 保存动作应给出本地 demo 状态反馈。

### 2.4 用例 4：重点题记录和学习记忆状态

援引 `design_docs/03_design_brief.md`：用例 4 是“生成重点题记录并更新记忆”。

Web 承载方式：

- 不作为第一版顶层导航项。
- 在“上传材料子流程”保存确认结果后展示处理状态。
- 在“学习成果”视图中按学科展示重点题摘要和材料处理状态。
- 后续可增加 Hermes job / memory update 的状态提示。

功能：

- 展示已确认重点题是否已进入 Hermes 总结。
- 展示重点题所属学科。
- 展示重点题数量、复习优先级和关联材料。
- 展示“生成总结”或“等待 Hermes 总结”的状态。

第一版边界：

- 可以使用静态 JSON 或本地 demo 状态模拟 Hermes 生成结果。
- 不要求真实实现长期 memory system。
- 不要求真实后端 job queue。

### 2.5 用例 5：周报生成状态和阶段性分析

援引 `design_docs/03_design_brief.md`：用例 5 是“生成阶段性学习分析和周报”。

Web 承载方式：

- 主要体现在 `历史周报` 视图。
- 第一版可以展示已经生成的静态 weekly report JSON。
- 后续可以增加“手动生成本周周报”入口，但不作为第一版必须交互。
- 周报采用跨学科总周报结构，而不是每个学科生成完全独立的周报。

功能：

- 展示周报生成状态。
- 展示跨学科整体学习状况。
- 按语文、数学、英语展示已上传材料、重点题记录、主要问题分析、知识点薄弱环节、复习建议和下周行动计划。
- 支持从学习成果或重点题确认结果进入对应周报。

第一版边界：

- 可以用静态 JSON 模拟每周生成结果。
- 不要求真实定时任务。
- 不要求真实 Hermes job runner。

### 2.6 用例 6：历史周报视图

援引 `design_docs/03_design_brief.md`：用例 6 是“Web 前端查询和展示周报”。

Web 承载方式：

- 顶层导航项：`历史周报`
- 读取 `/data/week_reports/week_reports_index.json`
- 用户选择某个周报后，再读取对应 `week_YYYYMMDD_YYYYMMDD.json`
- 该视图需要支持 `全部 / 语文 / 数学 / 英语` 的筛选器，用于在跨学科周报内查看整体或单学科内容。

功能：

- 查看历史周报。
- 切换周报周期。
- 查看跨学科总览。
- 查看各学科本周总结、上传资料、重点题、学习建议和下周行动。

主要内容：

- 周报选择器。
- 学科筛选器。
- 学生信息。
- 跨学科学习概览。
- 本周总结。
- 按学科组织的重点题列表。
- 按学科组织的上传资料列表。
- 按学科组织的学习建议。
- 下周行动。

## 3. 页面关系

学习成果主路径：

```text
学习成果
  -> 点击“上传材料”
  -> 上传材料子流程
  -> 自动切题完成
  -> 进入重点题确认步骤
  -> 保存确认结果
  -> Hermes 生成总结
  -> 历史周报展示
```

文字备注路径：

```text
打开 Hermes Web
  -> 输入备注
  -> 填写课堂提醒、疑问或复习备注
  -> 点击“保存备注”
  -> 备注进入本周学习上下文
```

历史周报也可以独立进入：

```text
打开 Hermes Web
  -> 历史周报
  -> 选择某一周
  -> 查看总结和建议
```

移动端：

```text
顶部栏:
  产品名
  学生姓名

底部或顶部标签导航:
  成果
  备注
  周报
  内容

主内容:
  单列布局
  学科筛选器可以放在主内容顶部或当前视图标题下方
```

上传材料子流程在桌面端可以使用全屏工作区；移动端应优先采用分步骤单列流程，避免原图、题目列表和确认表单互相挤压。

## 5. 默认视觉风格

- 清爽、可信、专业，适合教育科技场景。
- 信息密度适中，偏工作台，而不是宣传页。
- 使用克制的颜色、清晰的层级、稳定的间距。
- 可以使用图标辅助识别，但不要用装饰性大图抢占信息空间。
- 不使用过度装饰、夸张渐变或营销式排版。
- 中文内容不能溢出容器。

## 7. 页面生成指令模板

开发者可以使用下面模板向 Codex 描述目标页面的结构和风格。

```text
请使用 Vite + React + Tailwind CSS 实现 Hermes Web 学习助手工作台。

页面目标:
- 目标用户: 家长、学生、老师或项目评审者
- 核心任务: 按学科管理学习成果，上传试卷或错题本图片，确认需要记录的重点题，输入学习备注，并查看跨学科历史周报、学习总结和建议

学科设计:
- 第一版默认支持语文、数学、英语
- 内部 subject code 使用 chinese、math、english
- 顶层导航不按学科拆分
- 学科作为筛选器、表单字段和内容分区出现在学习成果、上传材料、输入备注、历史周报和学习内容视图中
- 周报采用跨学科总周报，页面内展示整体总结和各学科分区

数据来源:
- 演示阶段使用模拟 JSON 数据
- 上传材料后读取 /data/question_sessions/{upload_id}/question_split_result.json 展示切题结果
- 用户确认重点题后保存 /data/question_sessions/{upload_id}/question_confirmation_result.json
- 周报视图读取 /data/week_reports/week_reports_index.json
- 用户选择某个周报后，再读取对应 week_YYYYMMDD_YYYYMMDD.json

页面/视图:
- 学习成果视图
- 上传材料子流程
- 输入备注视图
- 历史周报视图
- 学习内容视图

默认布局:
- 不做营销首页，不做大幅首屏宣传区
- 第一屏直接展示可用的学习助手工作台
- 顶部栏显示产品名、学生姓名、当前周报周期和状态
- 左侧区域显示功能导航：学习成果、输入备注、历史周报、学习内容
- 主内容区域根据当前任务显示学习成果、备注输入、周报查看或学习内容
- 学习成果视图显示“添加材料”区域、最近材料、识别状态和重点题摘要
- “上传材料”按钮进入上传材料子流程
- 学习成果视图提供学科筛选器或学科分区
- 上传材料子流程显示学科选择、图片上传、识别状态、原图预览、题目框叠加、切题列表和确认表单
- 输入备注视图显示学科选择、文本输入框和明确的“保存备注”按钮
- 周报查看区显示跨学科学习概览、本周总结、按学科组织的上传资料列表、重点题、学习建议和下周行动

交互要求:
- 支持上传试卷或错题本图片
- 上传材料、教材 PDF 和备注时必须选择学科
- 支持按学科筛选学习成果、学习内容和周报内容
- 支持展示腾讯云切题结果
- 支持在原图上叠加题目框
- 支持选择需要记录的题
- 支持为选中题填写得分、满分、知识点、错因和备注
- 支持输入文字备注并通过按钮保存本地 demo 状态
- 支持切换不同历史周报
- 选择周报后更新页面内容
- 空数据、加载中、读取失败都要有基本状态

响应式要求:
- 桌面端可以采用左侧导航 + 主内容布局
- 移动端改为上下布局或标签布局
- 上传材料子流程在移动端采用分步骤单列布局
- 中文内容不能溢出容器
- 按钮、卡片、列表在窄屏下不能互相覆盖

工程要求:
- 拆分为清晰的 React 组件
- 先使用模拟 JSON 文件或组件内模拟数据
```
