# Demo JSON Contracts

This folder defines the first-version JSON contracts shared by Hermes, demo data, and Hermes Web.

The files are example-shaped contracts rather than strict JSON Schema files. They use stable field names, public URL-style paths, and sanitized demo values so the Web UI can be built before a real backend exists.

## Public URL Rules

- Use `/data/...` URLs for files that the Web UI may fetch or display.
- Do not store local absolute paths such as `/var/lib/hermes/data/...` in public JSON.
- Keep private/raw storage paths in backend configuration, not in these contracts.

## First-Version Contracts

- `textbook_content_summary.contract.json`: Hermes understanding of a textbook, chapter, or prepared textbook summary.
- `text_note.contract.json`: free-form student notes, knowledge points, reminders, or questions.
- `upload_meta.contract.json`: metadata for a material upload.
- `question_split_result.contract.json`: normalized result from question splitting/OCR.
- `question_confirmation_result.contract.json`: human-confirmed key-question records.
- `focus_question_records.contract.json`: Hermes-generated focus-question records.
- `week_reports_index.contract.json`: index file for historical weekly reports.
- `weekly_report.contract.json`: full weekly report detail consumed by Hermes Web.
- `learning_findings.contract.json`: local findings generated by `learning_insight_update` from per-material or per-event evidence. Each finding is local, traceable, confidence-annotated, and carries memory/action/weekly-context candidates.
- `insight_consolidation.contract.json`: consolidated insights formed by merging multiple local findings, short-term memory, and historical patterns. First phase consolidation runs inside `weekly_report` but keeps its own data structure.
- `learning_memory_snapshot.contract.json`: short-term or long-term memory snapshot recording stable patterns, active followups, and their source findings. Short-term memory absorbs weekly findings loosely; long-term memory requires stricter conditions.

## Evidence → Finding → Insight → Memory Pipeline

```
Evidence (upload, split, confirmation, notes, textbook summary)
  → Local Finding (per-material, per-event, confidence-annotated)
  → Short-term Memory (weekly findings, pending followups)
  → Consolidated Insight (multi-finding, multi-week patterns)
  → Long-term Memory (stable, recurring, confirmed patterns)
  → Weekly Report (cross-subject summary + action plan)
```

Key naming boundaries:
- **finding / 发现（局部发现）**: from a single upload, a key question, a note, or a small group of related materials. Traceable, local, confidence-annotated.
- **insight / 见解**: formed only through consolidation — merging multiple findings, short-term memory, and historical patterns. Cannot come from a single isolated event.
- **memory / 记忆**: preserved context. Short-term memory loosely absorbs weekly findings. Long-term memory requires recurrence, confirmation, or foundational impact.

First phase: consolidation lives inside `weekly_report` as a separate step (`weekly_report.consolidation_step`), with its own data structure so it can be extracted into a standalone `finding_consolidation` job later.

## ID Conventions

- `student_id`: stable student identifier, demo value is `小明`.
- `textbook_id`: stable textbook or material identifier.
- `upload_id`: upload session identifier, for example `upload_20260518_001`.
- `question_id`: question identifier scoped to an upload.
- `focus_question_id`: Hermes-generated focus-question record identifier.
- `weekly_report_id`: week identifier, for example `week_20260518_20260524`.

## Subject Conventions

All first-version learning materials must carry a `subject` field. Use stable internal subject codes in JSON, and use `subject_label` only for Chinese display text.

- `chinese`: 语文
- `math`: 数学
- `english`: 英语

Hermes Web should not use subject as a top-level navigation item. It should use `subject` for filtering, grouping, and rendering subject sections inside the four main views.
