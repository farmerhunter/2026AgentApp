# 设计简报

项目：面向初中学习的 AI 学习智能体

报名方向：2026 北京青少年人工智能应用实践活动 - AI 智能体应用

## 1. 项目定位

本项目演示一个主动式学习智能体 Hermes。它面向初中学生的日常学习场景，帮助学生把语文、数学、英语等多学科的试卷、错题本、作业、笔记和学习问题转化为可执行的复习行动。

第一版不是完整的生产级教育平台，而是一个可演示、可解释、能体现智能体价值的比赛作品。

核心价值：

- 从学生上传的学习材料中提取有用信息。
- 对所有学习材料记录学科，并支持跨学科汇总和按学科查看。
- 通过人机协作确认真正需要记录的重点题。
- 由 Hermes 总结各学科错因、知识点、复习建议和下周行动。
- 在 Web 前端中展示跨学科周报、各学科重点题和学习建议。

## 2. 目标用户

- 学生：上传试卷或错题本，确认需要记录的重点题，查看复习建议。
- 家长：了解本周学习情况、薄弱点和下一步计划。
- 老师：参考学生错题和知识点分布，了解学习进展。
- 比赛评委：观察 AI 智能体如何完成材料处理、分析、总结和行动建议闭环。

## 3. 第一版核心体验

第一版核心体验由几个可演示用例组成。它们共同说明 Hermes 不只是回答问题，而是能持续理解学习内容、接收学习成果、整理重点题、更新学习记忆，并生成阶段性分析和建议。

### 用例 1：上传课本 PDF，建立学习内容理解

学生或家长在“学习内容”视图中按学科上传课本、讲义或教材 PDF。Hermes 对 PDF 进行分析，提取目录、章节、知识点和学习单元，形成对当前学习内容的基础理解，并保存为教材摘要 JSON。

这个用例用于支持后续学习进度比对：当学生上传作业、试卷或备注时，Hermes 可以把这些材料关联到对应章节和知识点，判断当前学习内容、已覆盖范围和可能的薄弱环节。

第一版应实现可演示的课本 PDF 上传、Hermes 分析和教材摘要保存流程。为了控制范围，第一版可以只支持单个脱敏样例 PDF 或少量页码范围，不要求完整处理所有教材版式、扫描质量和复杂目录结构。

### 用例 2：上传学习成果并标注重点题

学生在 Hermes Web 的“学习成果”视图中按学科管理作业、试卷、练习册页面或错题本照片。“学习成果”视图是材料和结果的汇总入口，其中“添加材料”区展示已有上传记录、处理状态和最近材料；用户点击“上传材料”按钮后，进入具体的上传子流程。

上传子流程负责完成文件选择、材料信息填写、试题分割、重点题标注和确认保存。系统调用腾讯云 `QuestionSplitOCR` 完成试卷切题和基础 OCR，Hermes Web 在该子流程中展示原图、题目框和切题列表。

学生或家长手动选择需要记录的重点题，并补充得分、满分、知识点、错因、备注和复习优先级。

这个用例的重点不是追求全自动批改，而是通过“自动切题 + 人工确认”保证重点题记录可信。

### 用例 3：输入文字形式的知识点和备注

学生可以在“输入备注”视图中自由输入文字内容，例如课堂重点、老师提醒、自己不理解的知识点、复习备注或临时学习问题。备注必须指定学科，并可以进一步关联章节或知识点。该视图需要提供明确的操作按钮，例如“保存备注”或“加入本周上下文”，避免只有输入框而没有提交动作。

Hermes 将这些文字内容作为学习材料的一部分，与课本章节、作业、试卷和重点题记录放在一起分析。

这个用例用于覆盖无法通过图片或 OCR 表达的学习上下文，让学生能够直接告诉 Hermes 自己的理解状态和困惑。

### 用例 4：生成重点题记录并更新记忆

Hermes 根据已上传的课本内容、学习成果、人工确认的重点题和文字备注，按照智能体 skill 中定义的策略生成重点题记录。

重点题记录用于替代传统错题本，不只是保存题目本身，还保存学科、知识点、错因、学生标注、复习优先级和推荐行动。

生成后，Hermes 更新学习记忆，记录学生当前学习进度、薄弱点、已确认重点题和后续跟进事项。

### 用例 5：生成阶段性学习分析和周报

Hermes 可以定期生成周报，也可以由学生、家长或演示者手动触发生成。

周报内容采用跨学科总览形式，覆盖语文、数学、英语等学科。周报既要给出整体学习状况，也要按学科展示已上传材料、重点题记录、主要问题分析、知识点薄弱环节、复习建议和下周行动计划。

第一版建议以“每周生成 + 手动触发”作为演示方式：既能展示智能体的主动性，也方便比赛现场控制演示节奏。

### 用例 6：Web 前端查询和展示周报

Hermes Web 前端读取周报索引和周报 JSON，支持查看历史周报、切换周报周期，并展示跨学科总览、各学科学习总结、重点题、上传材料、问题分析、学习建议和下周行动。

第一版使用静态 JSON 作为数据源，后续可迁移到数据库和 REST API，支持长期检索、多学生和更丰富的筛选能力。

### 后续交互入口：微信端

后续可以补充微信端交互，例如通过微信上传图片、发送文字备注、接收提醒和查看简版周报。

微信端不作为第一阶段 Demo 的必选项。第一版优先保证 Web 工作台、重点题确认和周报展示流程完整可演示。

## 4. 演示重点

- 主动分析：上传材料后，Hermes 能生成下一步分析和建议。
- 多学科组织：所有学习材料必须带学科信息，周报以跨学科总览和学科分区展示。
- 人机协作：腾讯云负责试卷切题，学生负责确认重点题和语义信息。
- 可解释输出：每个建议应能追溯到上传资料、重点题或人工确认记录。
- 周报展示：Web 前端展示本周总结、上传材料、重点题、学习建议和下周行动。
- 可演进架构：演示阶段用静态 JSON，后续可迁移到数据库和 REST API。

## 5. 第一版不做的内容

为保证演示可落地，第一版不承诺：

- 自动可靠识别老师红笔对勾、叉号、圈画和划线。
- 自动理解老师手写批注。
- 自动判断老师给分、扣分和扣分原因。
- 自动为全部题目生成准确知识点和错因。
- 支持多学生、多班级、权限系统和长期生产级数据治理。

这些能力可以作为后续阶段研究方向，但不应成为第一版演示成败的关键依赖。

## 6. Web 体验原则

Hermes Web 第一版应是工作台，而不是营销首页。

设计原则：

- 第一屏直接进入可操作的学习助手界面。
- 页面采用“学习成果 / 输入备注 / 历史周报 / 学习内容”四个主视图组织。顶层导航不按学科拆分；学科作为筛选器和内容分区出现在各视图中。学习成果视图包含添加材料入口；具体上传、试题分割和重点题确认放在上传子流程中完成。
- 视觉风格清爽、可信、专业，适合教育科技场景。
- 信息密度适中，重点突出，不使用夸张装饰和营销式排版。
- 中文内容不能溢出容器，移动端需要支持基本查看和确认流程。

页面草图、视图关系和组件设想记录在 `docs/07_website_design_note.md`。

## 7. 预期输出

第一版应产出：

- 带学科字段的课本、试卷、错题本或备注上传记录。
- 腾讯云试卷切题结果 JSON。
- 学生人工确认的重点题记录。
- Hermes 生成的重点题总结、错因总结和复习建议。
- 周报 JSON 和周报索引文件。
- 可部署的 Hermes Web 前端。
- 比赛报告、演示脚本和演示视频素材。

## 8. 评审展示角度

面向比赛评审时，演示应强调：

- AI 智能体不是只回答问题，而是能持续组织学习材料并推动下一步行动。
- 系统承认 OCR 和批改语义识别的边界，用人工确认保证结果可信。
- 前端、存储和部署方案具备从演示阶段走向长期系统的路径。
- 输出内容以周报和行动建议呈现，便于学生和家长理解。
