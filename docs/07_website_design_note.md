# Hermes Web 网站草图设计笔记

本文档用于记录开发者对 Hermes Web 页面、功能和页面关系的初步设计。它不是最终 UI 规范，而是给 Codex 或前端开发者使用的文本草图。

## 1. 网站目标

Hermes Web 第一版不是营销首页，而是一个可操作的学习助手工作台。

核心任务：

1. 上传试卷或错题本图片。
2. 查看腾讯云试卷切题结果。
3. 手动确认哪些题需要记录。
4. 补充错因、得分、知识点和备注。
5. 查看 Hermes 生成的周报、学习总结和建议。

## 2. 页面和视图草案

第一版建议采用单页应用，但内部包含多个主要视图。

```text
Hermes Web 学习助手工作台
  -> 上传试卷
  -> 重点题确认
  -> 历史周报
```

### 2.1 上传试卷视图

功能：

- 上传试卷、练习册或错题本图片
- 显示上传状态
- 显示 OCR / 切题任务状态
- 显示最近上传记录
- 进入重点题确认视图

主要内容：

- 图片上传区域
- 学生、科目、来源类型、日期等基础信息
- 识别进度
- 最近上传列表

### 2.2 重点题确认视图

功能：

- 展示原始试卷图片
- 在原图上叠加题目框
- 展示切题列表
- 选择需要记录的题
- 为选中题填写得分、满分、知识点、错因、备注和复习优先级
- 保存确认结果
- 触发 Hermes 总结

主要内容：

- 左侧或主区域：原图预览和题目框
- 右侧或下方：题目列表
- 单题卡片：题目图片、识别文字、选择状态、确认表单
- 操作按钮：保存、生成总结、进入周报

### 2.3 历史周报视图

功能：

- 查看历史周报
- 切换周报周期
- 查看本周总结、上传资料、重点题、学习建议和下周行动

主要内容：

- 周报选择器
- 学生信息
- 学习概览
- 本周总结
- 重点题列表
- 上传资料列表
- 学习建议
- 下周行动

## 3. 页面关系

推荐用户路径：

```text
上传试卷
  -> 自动切题完成
  -> 进入重点题确认
  -> 保存确认结果
  -> Hermes 生成总结
  -> 历史周报展示
```

历史周报也可以独立进入：

```text
打开 Hermes Web
  -> 历史周报
  -> 选择某一周
  -> 查看总结和建议
```

## 5. 默认视觉风格

- 清爽、可信、专业，适合教育科技场景。
- 信息密度适中，偏工作台，而不是宣传页。
- 使用克制的颜色、清晰的层级、稳定的间距。
- 可以使用图标辅助识别，但不要用装饰性大图抢占信息空间。
- 不使用过度装饰、夸张渐变或营销式排版。
- 中文内容不能溢出容器。

## 7. 页面生成指令模板

开发者可以使用下面模板向 Codex 描述目标页面的结构和风格。

```text
请使用 Vite + React + Tailwind CSS 实现 Hermes Web 学习助手工作台。

页面目标:
- 目标用户: 家长、学生、老师或项目评审者
- 核心任务: 上传试卷或错题本图片，确认需要记录的重点题，并查看历史周报、学习总结和建议

数据来源:
- 演示阶段使用模拟 JSON 数据
- 上传试卷后读取 /data/question_sessions/{upload_id}/question_split_result.json 展示切题结果
- 用户确认重点题后保存 /data/question_sessions/{upload_id}/question_confirmation_result.json
- 周报视图读取 /data/week_reports/week_reports_index.json
- 用户选择某个周报后，再读取对应 week_YYYYMMDD_YYYYMMDD.json

页面/视图:
- 上传试卷视图
- 重点题确认视图
- 历史周报视图

默认布局:
- 不做营销首页，不做大幅首屏宣传区
- 第一屏直接展示可用的学习助手工作台
- 顶部栏显示产品名、学生姓名、当前周报周期和状态
- 左侧区域显示功能导航：上传试卷、重点题确认、历史周报
- 主内容区域根据当前任务显示上传区、题目确认区或周报查看区
- 上传区显示图片上传、识别状态和最近上传记录
- 题目确认区显示原图预览、题目框叠加、切题列表和确认表单
- 周报查看区显示学习概览、本周总结、上传资料列表、重点题、学习建议和下周行动

交互要求:
- 支持上传试卷或错题本图片
- 支持展示腾讯云切题结果
- 支持在原图上叠加题目框
- 支持选择需要记录的题
- 支持为选中题填写得分、满分、知识点、错因和备注
- 支持切换不同历史周报
- 选择周报后更新页面内容
- 空数据、加载中、读取失败都要有基本状态

响应式要求:
- 桌面端可以采用左侧导航 + 主内容布局
- 移动端改为上下布局或标签布局
- 中文内容不能溢出容器
- 按钮、卡片、列表在窄屏下不能互相覆盖

工程要求:
- 拆分为清晰的 React 组件
- 先使用模拟 JSON 文件或组件内模拟数据
- 保留后续替换为真实 fetch JSON/API 的接口位置
- 样式使用 Tailwind CSS
- 第一版暂不引入 shadcn/ui 或图表库；如果后续引入，需要说明原因
- 完成后运行 npm run build 验证构建通过
```

## 8. 待继续设计的问题

- 重点题确认是否采用“原图 + 右侧列表”还是“题目卡片流”作为主布局。
- 移动端是否先只支持上传和基础确认，复杂编辑留给桌面端。
- 周报视图是否需要和重点题确认视图共用重点题卡片组件。
- 是否需要为评委准备一个只读演示模式。
